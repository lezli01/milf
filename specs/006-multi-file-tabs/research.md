# Phase 0 — Research: Multi-File Editing with Tabs

This document resolves the open technical questions for Feature 006. Each section captures the **decision**, the **rationale**, and the **alternatives considered**, so future contributors can re-open a choice with full context. Foundations laid by Feature 002 (CodeMirror 6 editor, markdown-it + DOMPurify preview, Tailwind CSS v4 islands aesthetic, prop-drilled React state), Feature 003 (Open file, view modes, theme, `localStorage` chokepoint in `preferences.ts`), and Feature 004 (Save, auto-save, `<FileHeader />`, `savedText`-derived modified flag, single chokepoint in `fileOpen.ts`, per-write `saving` mutex with a follow-up ref) are assumed; this feature reshapes that state from "one document" to "N documents on a strip".

## 1. Tab data model — flat array + ID pointer, IDs are a monotonic counter

**Decision**: Top-level state in `App.tsx` is `tabs: Tab[]` and `activeTabId: TabId | null`, where:

```ts
type TabId = string; // monotonic, e.g. "tab-1", "tab-2", "tab-3"…
type Tab = {
  id: TabId;
  text: string;          // the live editor text for this tab
  savedText: string;     // last successfully written content (or content at open time)
  openedFile: { name: string; path: string } | null; // null = untitled
};
```

Tab order is insertion order; the active tab is identified by `activeTabId` (not by a per-tab boolean, which would have to be kept consistent on every reorder/insert). The derived `activeTab` is `tabs.find(t => t.id === activeTabId) ?? null`. The current editor text the `<Workspace />` renders is `activeTab?.text ?? ""`.

`TabId`s are generated by a tiny in-file `nextTabId()` helper that maintains a module-scoped counter (`let counter = 0; () => "tab-" + (++counter)`). The counter is process-local and is reset on every app launch — that is acceptable because tabs are not persisted across launches (spec Assumptions).

**Rationale**:
- A flat array maps directly onto how the tab strip is rendered (`tabs.map(t => <TabPill />)`). Insertion order = visible order, no separate "order" field needed.
- An `activeTabId` pointer (rather than `isActive` on each tab) makes "switch the active tab" a single state update (`setActiveTabId(newId)`) instead of an N-way invariant maintained across every tab object on every switch. The classic two-active-tabs / no-active-tab bug becomes impossible by construction.
- Generating IDs with a monotonic counter avoids a UUID dependency and makes the IDs easy to read in React DevTools when debugging. Per-launch reset is fine because nothing references an ID across launches.
- Using the file path as the tab ID was considered (and rejected — see Alternatives) because untitled tabs have no path and Save-As changes a tab's path mid-life. The path is still the key for **dedup** (FR-011), but it is not the **identity**.

**Alternatives considered**:
- **`Map<TabId, Tab>` keyed on path or ID**: `Map` order is technically insertion order in modern engines, but accessing `tabs[i]` for the strip render becomes `Array.from(map.values())[i]` — three more characters of indirection on every render. Array wins.
- **Per-tab `isActive: boolean`**: requires every tab object to be updated on every switch. The pointer form is O(1); the boolean form is O(N).
- **Path-keyed identity**: untitled tabs have no path. Save-As changes a tab's path, which would orphan any React `key={path}` references and force re-creation. Path-as-ID also gets messy if two distinct opens of the same file land in the same render tick (rare but real with keyboard-shortcut spam) — a counter never collides.
- **UUID library (`crypto.randomUUID`, `uuid` package)**: `crypto.randomUUID` is available in modern WebView2 / WKWebView / WebKitGTK so a library dep is unnecessary, but the strings (36 chars) are also overkill for an in-memory counter that resets on launch. A short `"tab-N"` string is more debuggable and equally unique within a session.

**Implementation notes**:
- The counter lives at module scope in `App.tsx` (or a tiny `src/lib/tabId.ts` if it grows; not warranted at one call site). `const nextTabId = (() => { let n = 0; return () => `tab-${++n}`; })();`.
- The list of tabs is reset to `[]` only on close-last-tab (FR-018). There is no other path that empties it.
- `activeTabId` is set to `null` ONLY when `tabs.length === 0`. Whenever `tabs.length > 0`, exactly one tab is active (FR-002 made structural by the pointer model).

## 2. Dedup on Open — re-focus the existing tab, never reload from disk

**Decision**: When the user activates the Open control, look up the picked path against `tabs.find(t => t.openedFile?.path === pickedPath)`. If found, just `setActiveTabId(existingTab.id)` — do NOT re-read the file from disk, do NOT replace the existing tab's `text` or `savedText`. If not found, create a new tab (`tabs = [...tabs, newTab]`), activate it, and the editor `<Workspace />` automatically re-renders against the new active tab.

**Rationale**:
- Path-equality is the canonical "is this the same file?" identity. The Tauri dialog returns the user-resolved absolute path; two opens of the same dialog-selected file produce string-equal paths in 100% of cases.
- Re-focusing rather than reloading respects the user's in-memory edits — if they had unsaved work in that tab, a silent disk reload would discard it. The spec is explicit (Edge Cases: "the application MUST NOT silently reload from disk and discard the edits") and "reload from disk" is listed as a candidate follow-up.
- This makes the open path simpler: one branch (dedup hit) is a state update; the other branch (dedup miss) is the existing open flow with a new tab wrapped around it.

**Alternatives considered**:
- **Canonicalise paths first (case-fold on Windows, resolve symlinks)**: useful for case-insensitive filesystems but introduces platform branches we don't need yet. The Tauri dialog returns canonical paths on each platform; double-opening from the same dialog produces equal strings. Re-open this decision if real bug reports of "two tabs for the same file" show up.
- **Reload from disk on re-open**: silently discards unsaved edits. Rejected (see spec Edge Cases).
- **Prompt the user "this file is already open — refocus or open a copy?"**: dialog soup. The right behaviour is the obvious one — bring me back to the tab I had.
- **Use inode / file ID instead of path**: not portable across the three OSes via Tauri's plugin-fs surface; path equality is sufficient at this scale.

**Implementation notes**:
- The lookup is `tabs.findIndex(t => t.openedFile?.path === path)`. If `>= 0`, activate; otherwise append.
- Untitled tabs (`openedFile === null`) never collide on path — they have no path. They are de-duped only by ID (i.e., not at all).
- Dedup happens AFTER the open dialog returns and the path is known. If the read of the file fails (permission denied, vanished), no tab is created and the existing tabs are untouched — same error path as Feature 003 / 004.

## 3. CodeMirror state preservation across tab switches — snapshot via `EditorView.state` + `setState`

**Decision**: Add a `useRef<Map<TabId, EditorState>>(new Map())` in `App.tsx` (call it `editorStatesRef`). Expose a small imperative API on `<Editor />` via `forwardRef`:

```ts
type EditorHandle = {
  getState(): EditorState;
  setState(state: EditorState): void;
};
```

When the user switches the active tab from `A` to `B`:
1. `editorStatesRef.current.set(A, editorHandle.getState())` — snapshot outgoing state.
2. `setActiveTabId(B)` — React re-renders with the new active tab's text as `value`.
3. In a `useLayoutEffect` keyed on `activeTabId`: if `editorStatesRef.current.has(B)`, call `editorHandle.setState(editorStatesRef.current.get(B)!)` — restore selection, scroll, and undo history. Otherwise (first time the tab is activated), do nothing — the freshly-mounted `EditorView` already has the right `doc` because we pass `value={activeTab.text}`.

`EditorState` is CodeMirror's first-class immutable state object; it carries the doc, the selection ranges, the scroll snapshot from `EditorView.scrollSnapshot()` (rolled into `EditorState` for free via standard extensions in our config), and the undo history (because we already include the `history()` extension). `view.setState(...)` swaps the entire state atomically in one transaction. There is no half-mounted intermediate frame.

**Rationale**:
- This is the canonical CodeMirror 6 pattern for "swap which document an `EditorView` is showing while keeping per-document state alive". The CodeMirror 6 architecture is built around `EditorState` being a serialisable snapshot for exactly this scenario.
- One `EditorView` is alive at any time (cheap), N `EditorState`s sit in a `Map` (each is a few KB of structured data — CodeMirror's `Text` is a persistent rope, not a flat string copy). For 10 tabs of 100 KB, that's ~100 KB of doc data + ~10 lightweight `EditorState` shells — well below desktop noise.
- Keeping the map in a `useRef` (not React state) is intentional: these objects are large and React would shallow-compare them on every render. Refs are exactly the right tool for "things that survive across renders but never trigger them".
- A `useLayoutEffect` (not `useEffect`) runs synchronously after DOM commit but before the browser paints — so the user never sees a frame of the wrong tab's content with the wrong cursor.
- The "first activation has no snapshot" branch correctly handles the case where a tab was just created (Open → new tab → activated). The Editor's existing "sync `value` prop to doc" effect handles the initial population; on the next switch we have a snapshot.

**Alternatives considered**:
- **One `<Editor />` per tab, hidden via `display: none` when inactive**: works but mounts N `EditorView`s — N × CodeMirror init cost on startup, N × cleanup on close-all. For 10 tabs that's a noticeable startup hiccup. Also breaks the existing "Editor stays mounted across view-mode switches" invariant from Feature 003 §3 by introducing a *second* mount/unmount axis (per-tab, in addition to per-view-mode).
- **Re-mount Editor on every tab switch (`<Editor key={activeTabId} />`)**: loses cursor and scroll entirely on every switch — fails FR-001 ("best-effort cursor and scroll").
- **Persist only `text` + cursor offset per tab, restore by dispatching a transaction**: loses scroll position and undo history. The CodeMirror docs and ecosystem (and most editors that have multi-doc tabs — e.g., the CodeMirror playground itself) use full `EditorState` swap precisely for this reason.
- **Store `EditorState` snapshots in React state**: forces React to shallow-compare large structured objects on every render. Refs are the textbook choice for "mutable across renders, not part of render output".

**Implementation notes**:
- `<Editor />` becomes `forwardRef<EditorHandle, EditorProps>(({ value, onChange }, ref) => { … })`. Inside, `useImperativeHandle(ref, () => ({ getState: () => viewRef.current!.state, setState: (s) => viewRef.current!.setState(s) }), [])`.
- The existing `useEffect` in Editor that re-syncs `value` into the doc still runs on every prop change. To avoid clobbering a freshly-restored state, gate that effect: only dispatch the doc-replace transaction if `view.state.doc.toString() !== value`. (This is already the existing implementation — see `src/components/Editor.tsx:88-93` — so no change needed.)
- `App.tsx` calls `editorStatesRef.current.set(prevId, editorRef.current!.getState())` inside the same handler that calls `setActiveTabId(nextId)`, BEFORE the state update. Then in a `useLayoutEffect` keyed on `[activeTabId]`, it calls `editorRef.current!.setState(snapshot)` if a snapshot exists.
- When a tab is closed, remove its entry from `editorStatesRef.current` (`editorStatesRef.current.delete(closedId)`) to free the snapshot. This is a tiny memory hygiene win, not a correctness requirement (the JS engine would GC it once the map entry is gone).

## 4. Per-tab `saving` flag + per-tab `pendingSaveRef` — Maps, not scalars

**Decision**: Replace the single `saving: boolean` state and the single `pendingSaveRef: { current: boolean }` from Feature 004 with per-tab `Map`s:

```ts
const [savingByTab, setSavingByTab] = useState<Record<TabId, boolean>>({});
const pendingSaveRef = useRef<Map<TabId, boolean>>(new Map());
```

`performSave(tabId: TabId)` looks at `savingByTab[tabId]` and `pendingSaveRef.current.get(tabId)` — same algorithm as Feature 004 §3 but scoped to the given tab. The Save button reads `savingByTab[activeTabId] ?? false`. The auto-save effect watches the active tab specifically; its dependency array includes the active tab's text, savedText, and savingByTab[activeTabId].

`savingByTab` uses a plain `Record` (not `Map`) because React needs immutable updates to detect state changes, and `{...prev, [id]: true}` is the idiomatic React pattern for keyed state. `pendingSaveRef` uses a `Map` because it lives in a ref and mutates in place — no React subscription is needed.

**Rationale**:
- The close-tab-with-Save path (FR-016) operates on a non-active tab. With a single global `saving` flag, the active tab's auto-save would block (FR-013 requires it to fire) just because a different tab is being closed-and-saved. Per-tab flags isolate the two operations.
- This is also the structural answer to SC-004 ("write affects only the active tab's file in 100% of cases"). With per-tab flags, a save kicked off for tab X literally cannot mutate tab Y's `savedText` — there is no shared mutable.
- Per-tab `pendingSaveRef` preserves the "coalesce concurrent triggers into at most one follow-up" behaviour from Feature 004 §3 — independently for each tab.
- The two collections together are O(open-tab-count) — 10 booleans + 10 booleans for SC-001's 10-tab budget. Trivial.

**Alternatives considered**:
- **One global `saving` flag**: races as described above. Rejected.
- **A queue of pending saves per tab**: over-engineered. The "at most one follow-up after the current write" idiom from Feature 004 is the simplest correct serialisation, applied independently per tab.
- **A `Promise`-chain mutex per tab** (`Map<TabId, Promise<void>>`): correct but harder to reason about than a flag + ref. Save throughput is not the bottleneck; clarity is.
- **Drop `pendingSaveRef` entirely and let the auto-save effect re-fire on the next keystroke**: would lose the "type a character during the in-flight save → that character also reaches disk" guarantee. Acceptable for auto-save (it'll fire again soon) but the close-tab Save path needs a one-shot guarantee — the user clicked Save in the close-confirm dialog and the tab is about to be closed; there is no "next debounce".

**Implementation notes**:
- `performSave(tabId)` body sketch (drawn from Feature 004 §3, scoped per tab):
  ```ts
  async function performSave(tabId: TabId) {
    if (savingByTab[tabId]) {
      pendingSaveRef.current.set(tabId, true);
      return;
    }
    setSavingByTab(prev => ({ ...prev, [tabId]: true }));
    const tab = tabs.find(t => t.id === tabId);
    if (!tab) return; // tab was closed mid-save; the close path handles cleanup
    const outbound = tab.text;
    const result = tab.openedFile === null
      ? await saveMarkdownFileAs(outbound, "Untitled.md")
      : await saveMarkdownFile(tab.openedFile.path, outbound);
    if (result.kind === "ok") {
      updateTab(tabId, t => ({
        ...t,
        savedText: outbound,
        openedFile: t.openedFile ?? { name: result.name!, path: result.path! }
      }));
      setError(null);
    } else if (result.kind === "error") {
      setError(`Could not save ${tabTitle(tab)}: ${result.message}`);
    }
    setSavingByTab(prev => ({ ...prev, [tabId]: false }));
    if (pendingSaveRef.current.get(tabId)) {
      pendingSaveRef.current.set(tabId, false);
      queueMicrotask(() => void performSave(tabId));
    }
  }
  ```
- The error message is prefixed with the tab title (basename or `Untitled-N`) so FR-021's attribution requirement is met: the user can tell which tab's save failed.
- When a tab is closed, clear its entries: `setSavingByTab(prev => { const next = {...prev}; delete next[id]; return next; })` and `pendingSaveRef.current.delete(id)`. Stale `true` values in the Map would block future save attempts on a recycled (re-opened) tab ID — though IDs are monotonic so this is theoretical.

## 5. Auto-save effect — still one `useEffect`, now keyed on the active tab

**Decision**: The single auto-save `useEffect` from Feature 004 is preserved but its dependency array switches from `[text, savedText, autoSave, openedFile, saving]` to `[activeTabId, activeTab?.text, activeTab?.savedText, autoSave, activeTab?.openedFile, savingByTab[activeTabId ?? ""]]`. The effect's body is the same shape — if the active tab is eligible (auto-save on, openedFile non-null, modified, not currently saving), schedule a `setTimeout` that calls `performSave(activeTabId)` after 1500 ms. The cleanup function clears the timer.

If the user switches the active tab during the debounce window, the effect re-runs because `activeTabId` changed; the cleanup cancels the timer for the previous tab. No save fires on the previous tab — that matches FR-020 (auto-save acts on the **active** tab; switching away means the user isn't typing in the previous tab anymore). The previous tab keeps its modified indicator and the user can save it manually.

**Rationale**:
- Same `useEffect` shape, same trailing-edge debounce, same 1500 ms midpoint as Feature 004 §2 — just scoped to the active tab. There is no new debounce library, no new timer pattern.
- "Switching tabs cancels the pending auto-save for the previous tab" is the simplest reading of FR-020. The spec says auto-save fires on idle of the active tab; the moment the user switches, the active tab is different and the previous tab is no longer being "typed into". The modified indicator on the previous tab makes the cancelled state visible — the user knows what's unsaved.
- An untitled active tab is NOT auto-savable (its `openedFile === null`). FR-014 from Feature 004 explicitly says "while auto-save is enabled but no file is currently opened, the application MUST NOT write anywhere". Per-tab this becomes "while the active tab has no backing file, the auto-save effect skips". The effect's existing `if (openedFile === null) return` line handles this unchanged.

**Alternatives considered**:
- **Per-tab debounce timers stored in a ref Map**: lets all dirty tabs auto-save in parallel after their own idle windows. Reads more aggressive than the spec wants — FR-020 says auto-save acts on the active tab. Also creates a footgun: if the user types in 3 tabs in quick succession, 3 saves race. Strict-active-tab is simpler and matches the spec.
- **Auto-save the previous tab on tab switch**: tempting but not what the spec asks for. FR-020 explicitly says auto-save fires on idle, and a tab switch is not "idle in the previous tab" — it's the user moving on. Listed as a candidate follow-up in spec Assumptions.
- **Drop the auto-save effect entirely and rely on tab close-prompts to catch unsaved work**: regresses Feature 004 — auto-save is a documented feature. Out of the question.

**Implementation notes**:
- The effect's `if` chain becomes:
  ```ts
  useEffect(() => {
    if (!autoSave) return;
    const active = tabs.find(t => t.id === activeTabId);
    if (!active) return;
    if (active.openedFile === null) return;
    if (active.text === active.savedText) return;
    if (savingByTab[active.id]) return;
    const id = setTimeout(() => void performSave(active.id), 1500);
    return () => clearTimeout(id);
  }, [activeTabId, /* derived from active: */ tabs, autoSave, savingByTab]);
  ```
  (The dep array includes `tabs` because the active tab's `text` / `savedText` / `openedFile` live inside `tabs`; React's shallow-compare on the array reference is enough — every state update produces a new `tabs` array via `setTabs`.)
- The `eslint-disable-next-line react-hooks/exhaustive-deps` comment from Feature 004 may carry forward to silence the same `performSave` warning. The function is recreated each render and reads the latest `tabs` / `savingByTab` via closure.

## 6. Close-tab confirmation — native `<dialog>` element + three buttons

**Decision**: Implement the close-with-unsaved-changes confirmation as a tiny `<ConfirmDialog />` component that wraps the browser-native `<dialog>` element. The component takes the three actions as props:

```ts
type ConfirmDialogProps = {
  open: boolean;
  title: string;
  message: string;
  onSave: () => void;
  onDiscard: () => void;
  onCancel: () => void;
};
```

The component renders three buttons in tab order Save → Discard → Cancel, with Save autofocused (the default destructive-but-safer choice). Pressing ESC dismisses to Cancel — `<dialog>` fires the `cancel` event natively. Clicking the backdrop does NOT dismiss (we want the user to make an explicit choice — `onCancel` is reached via the Cancel button or ESC).

**Rationale**:
- The native `<dialog>` element provides modality, focus trapping, ESC-to-close, and top-layer rendering for free — all of which a custom `<div role="dialog">` would have to reimplement. WebView2, WKWebView (macOS 12+ via WebKit recent enough), and modern WebKitGTK all ship `<dialog>` with `showModal()`. The constitution's cross-platform principle is satisfied by built-in browser features rather than a portal/focus library.
- Three explicit buttons cover FR-015's "at least three choices". The button order — Save first, Cancel last — matches the "default action is the safest" convention used by VS Code, Sublime, TextEdit, Notepad++.
- No new runtime dep. The component is ~40 lines including the buttons, styling, and the dialog `open`/`showModal()` plumbing. Tailwind handles the islands styling identically to `<ErrorBanner />` and the toolbar.
- `window.confirm()` was explicitly considered and rejected: it offers only OK and Cancel, which cannot express the three-way choice. Forcing the user through two sequential `confirm()` calls ("Save? Y/N", "Discard? Y/N") is worse UX.

**Alternatives considered**:
- **`window.confirm()`**: two buttons only. Rejected.
- **A modal library (`react-modal`, `@radix-ui/react-dialog`, `react-aria` Dialog)**: new runtime dep, focus-trap baggage we don't need on a single dialog. Rejected on Principle I + dependency justification.
- **A custom `<div role="dialog">` with portal + focus trap**: re-implements what `<dialog>` provides natively. Rejected — see "first principles" justification above.
- **An inline confirmation strip (no modal)**: visually awkward — the tab being closed is at the top of the strip; the prompt would be elsewhere; user has to look in two places to act.

**Implementation notes**:
- `<ConfirmDialog />` uses a `useEffect` to call `dialogRef.current!.showModal()` when `open` becomes true and `dialogRef.current!.close()` when it becomes false.
- The `cancel` event (ESC press) is wired to `onCancel`.
- Tailwind classes match the existing `<ErrorBanner />` / `<FileHeader />` / `<Toolbar />` islands aesthetic: rounded-2xl, ring-1, backdrop-blur, the same surface/ring color variables.
- The close-tab handler in `App.tsx` flow:
  ```ts
  function handleCloseTab(id: TabId) {
    const tab = tabs.find(t => t.id === id);
    if (!tab) return;
    const isModified = tab.text !== tab.savedText;
    if (!isModified) { removeTab(id); return; }
    setPendingClose(id); // triggers <ConfirmDialog open={...} />
  }
  function onConfirmSave()    { void performSaveThenClose(pendingClose); }
  function onConfirmDiscard() { removeTab(pendingClose); setPendingClose(null); }
  function onConfirmCancel()  { setPendingClose(null); }
  ```
- `performSaveThenClose(id)` awaits `performSave(id)`; if `setError` was set as a result, the close is aborted (tab stays open, error is visible) — exactly FR-016's "on save failure the tab and its edits remain". The straightforward implementation reads the error state after the save returns; if no new error is set, the close proceeds.

## 7. Neighbor activation on close-active-tab

**Decision**: When the active tab is closed AND at least one other tab remains, activate the **right neighbor** of the closed tab. If the closed tab was the rightmost, activate the **left neighbor**. Implementation: compute the index of the closing tab, remove it, then set the new active to `tabs[oldIndex]` (which is now the right neighbor since we shifted left) or to `tabs[oldIndex - 1]` if `oldIndex` is now out of bounds.

**Rationale**:
- Right-neighbor-first matches the convention in VS Code, Sublime Text, browsers (Chrome, Firefox), and macOS Finder tabs. Users coming from any of those have the right muscle memory.
- The two-line implementation is bulletproof: `const idx = tabs.findIndex(t => t.id === id); const next = tabs[idx + 1] ?? tabs[idx - 1] ?? null; setActiveTabId(next?.id ?? null);`.
- FR-017 demands a "sensible neighbor" without prescribing a specific direction; right-first is the broadest convention.

**Alternatives considered**:
- **Most-recently-used (MRU) order**: requires a separate stack of recently-visited tab IDs and a fair bit of bookkeeping. Worth it in IDEs with hundreds of tabs; overkill for a 10-tab markdown editor.
- **Always activate the first (leftmost) tab**: jumps the user's attention unpredictably. Rejected.
- **Always activate the previously active tab before the closed one**: ambiguous semantics (what if there isn't one?), and breaks if the user closes several tabs in a row.

**Implementation notes**:
- The same logic handles "close the only tab" (next is `null` → activeTabId becomes `null` → empty state per FR-018).
- The snapshot map is also cleaned up — `editorStatesRef.current.delete(closedId)` — in the same `removeTab` function.

## 8. Empty state when zero tabs are open

**Decision**: When `tabs.length === 0`, the workspace presents:
- The TabStrip rendered as an empty island (the rounded-2xl ring band stays visible for visual consistency, but contains no tab pills — optional: a faded "No files open" label inside).
- The Editor pane renders an empty CodeMirror buffer (still mounted, `value=""`).
- The Preview pane renders an empty rendered output (also mounted in split / preview-only modes).
- The Save button is disabled (`saveEnabled = activeTabId !== null && active.openedFile !== null && !savingByTab[active.id]`).
- The auto-save checkbox remains visible and its setting remains intact (per Feature 004 FR-014).

**Rationale**:
- FR-003 explicitly requires the empty state to mirror "no file opened" behaviour from Features 003 / 004. The chosen implementation simply makes the existing Save-button-disabled / Editor-empty / Preview-empty branches the natural result of `activeTabId === null`.
- Keeping the Editor and Preview mounted (even with empty content) means the user clicking "New" or "Open" populates them without a mount delay — same UX continuity that Feature 003 §3 established.
- Hiding the TabStrip entirely when empty was considered but rejected: a band that appears and disappears as tabs come and go is visual noise. Rendering it always (with empty contents) keeps vertical layout stable.

**Alternatives considered**:
- **Auto-create an Untitled tab on launch / on close-all so there's always at least one tab**: more aggressive than the spec wants. Spec FR-018 explicitly says close-last-tab returns to the empty state — not "creates an untitled tab".
- **Hide TabStrip entirely when empty**: the workspace's vertical layout shifts every time the strip appears or disappears. Visually noisy.
- **Show a centered "Welcome / Open a file" CTA in the editor area when empty**: nice-to-have, not required by spec, and would be a fourth component. Defer to a follow-up.

**Implementation notes**:
- The TabStrip component handles the empty case internally (`if (tabs.length === 0) return <div className={emptyClass}>No files open</div>;`).
- `saveEnabled` becomes `activeTabId !== null && active.openedFile !== null && !savingByTab[active.id]` — same shape as Feature 004's `saveEnabled = openedFile !== null && !saving`, just per-tab.
- The keyboard shortcut for Save (Ctrl/Cmd+S) is a no-op when `saveEnabled === false` — the existing handler already checks the disabled flag through `performSave`'s own guards.

## 9. Tab strip overflow — horizontal scroll with `overflow-x-auto`

**Decision**: When the tab strip cannot fit all tabs at the current window width, it scrolls horizontally. Tailwind `overflow-x-auto` plus `flex-nowrap` on the inner row gives this for free. Tab widths are capped (`max-w-[180px]` per tab) so a single very-long-named tab cannot push everything off-screen; long names are truncated within each pill (the existing Feature 004 truncation pattern applied per-pill instead of to one header line). The full file name + path is reachable via the tab's `title` attribute on hover.

**Rationale**:
- Horizontal scroll is the most predictable overflow pattern — every desktop OS user has scrolled a horizontal strip (browser tabs, file managers, IDE tabs). It does not hide tabs behind a menu, it does not collapse them into an unfamiliar UI affordance.
- `overflow-x-auto` shows a scrollbar only when needed; on a wide window the strip lays out normally with no scrollbar. The scrollbar styling inherits from the OS — consistent across the three platforms.
- The active tab can be auto-scrolled into view when activated (`scrollIntoView({ behavior: 'instant', block: 'nearest', inline: 'nearest' })`) so the user never has the active tab off-screen after a switch.
- Per-pill truncation handles the "very long filename in one tab" case (FR-008) without affecting other tabs. The `title` attribute provides progressive disclosure for both the file name and the full path.

**Alternatives considered**:
- **Wrap to multiple rows**: dynamically variable vertical space breaks the layout's vertical math. Rejected.
- **Collapse off-screen tabs into a "more…" dropdown menu**: more code (a dropdown, keyboard nav, focus management) for a feature with at most ~20 tabs in practice. Defer to a follow-up if real user reports demand it.
- **Shrink tab widths to fit all tabs**: at 30 tabs each pill becomes unreadable. Capping per-pill width and scrolling preserves legibility.
- **A virtualised list**: massive over-engineering for double-digit tab counts.

**Implementation notes**:
- The TabStrip's outer container: `flex items-center gap-1 overflow-x-auto px-2`.
- Each pill: `flex items-center gap-1.5 px-2 py-1 min-w-0 max-w-[180px] rounded-md ring-1 cursor-pointer select-none` plus active-state classes.
- The truncated label inside the pill: `truncate flex-1 min-w-0` with `title={fullPath ?? displayName}` on the pill itself.
- Active-tab auto-scroll: a `useEffect` in TabStrip keyed on `activeTabId` finds the active pill's DOM node (via a `Map<TabId, HTMLElement | null>` ref or `data-active="true"` selector) and calls `scrollIntoView`. ~5 lines.

## 10. Per-tab modified indicator vs FileHeader removal

**Decision**: Each tab displays an asterisk prefix (`*`) when its `text !== savedText`. The standalone `<FileHeader />` component file is deleted in the same PR. The window title (Feature 003 FR-007) is preserved and continues to reflect the active tab's file name — `setWindowTitle(activeTab?.openedFile?.name ?? null)` runs in a `useEffect` keyed on `[activeTabId, activeTab?.openedFile?.name]`.

**Rationale**:
- FR-007 (per-tab modified indicator) is satisfied by an asterisk on each tab's title — same convention as `<FileHeader />` used for the single-tab case in Feature 004 §6. Users carry one mental model across the deletion + addition.
- FR-023 (delete the standalone header) is satisfied by deleting `src/components/FileHeader.tsx` and removing its import + render site in `App.tsx`. No "hide behind a flag" — the spec is explicit that it goes.
- FR-024 (window title unaffected) is satisfied by keeping the `setWindowTitle` call in `App.tsx`; the effect's dependency just changes from `[openedFile?.name]` to `[activeTabId, activeTab?.openedFile?.name]`.
- The modified indicator on an Untitled tab: the asterisk shows if `text !== savedText`. For a fresh Untitled tab (just created, content === "" === savedText === ""), no asterisk. As soon as the user types one character, asterisk appears. After Save-As, savedText becomes the typed content, asterisk clears.

**Alternatives considered**:
- **Keep `<FileHeader />` as a redundant secondary indicator**: violates FR-023. Rejected.
- **Show the asterisk inside the tab's close-button slot (replacing the × when modified)**: clever but hides the close affordance from users with unsaved work — exactly the wrong direction.
- **Show the asterisk only on the active tab**: makes it impossible to see which background tabs have unsaved changes — violates FR-007 ("when a tab's content differs… that tab MUST display"). Rejected.

**Implementation notes**:
- The tab pill renders `{isModified && <span aria-label="modified">*&nbsp;</span>}<span className="truncate">{title}</span><CloseButton />`.
- The close button is conditionally shown — every tab has one (FR-013). Visibility tuning (hover-only vs always) is a visual-design detail; the recommendation is always-visible for accessibility (keyboard users can't hover).
- After deleting `<FileHeader />`, `App.tsx`'s import block shrinks by one line and the render tree replaces `<FileHeader ... />` with `<TabStrip ... />` in the same slot.

## 11. Untitled tabs — minimal extension to keep the existing New button working

**Decision**: The codebase ships a New button (`handleNewFile` in `App.tsx`, from Feature 004). The spec's Assumption "tabs with no backing file are out of scope" was written without acknowledging this. Choices were: (a) regress the New button by removing/disabling it during this feature; (b) extend scope minimally to support untitled tabs. **Choice: (b).** New creates an `Untitled-N` tab with empty content and `openedFile: null`. The N suffix is the count of existing untitled tabs at creation time (so multiple untitled tabs are distinguishable in the strip).

Saving an untitled tab uses the existing `saveMarkdownFileAs(content, "Untitled.md")` path that `App.tsx` already calls when `openedFile === null` (this code path was added in Feature 004 and is still live). After Save-As succeeds, the tab's `openedFile` is populated with `{ name, path }` and its title in the strip becomes the new basename. Save / auto-save then behave as for any file-backed tab.

**Rationale**:
- Removing the New button would be a user-visible regression that warrants its own spec. The cleanest move is to acknowledge the existing UX and extend the spec's scope minimally to match.
- The Save-As-on-Untitled-Save path is already implemented in the codebase (see `src/App.tsx:91-103` in the current `performSave`). Re-using it costs essentially zero code — the per-tab `performSave` keeps the same branch.
- The `Untitled-N` numbering uses a simple "next available N" rule: scan existing untitled tabs in the current `tabs` array, find the highest N suffix, add 1. New windows / launches reset N to 1 (no persistence — see spec Assumptions).

**Alternatives considered**:
- **Strictly honour the spec assumption and remove/disable the New button**: regresses existing UX. Better to extend the spec (and document the extension in Complexity Tracking) than to silently shrink the user-facing feature set.
- **Keep New but make it replace the active tab's content (old behaviour adapted)**: silently wipes the active tab's content while other tabs survive — confusing and inconsistent. Rejected.
- **Treat Untitled tabs as ephemeral and forbid Save (only Save-As)**: spec FR-019 says Save acts on the active tab; Save on an Untitled tab via the existing Save-As fallback is the obvious correct behaviour. No reason to forbid it.

**Implementation notes**:
- `handleNewFile()` becomes "create a new tab with `text: ""`, `savedText: ""`, `openedFile: null`, and a name like `Untitled-1` for the title, then activate it". The Tab's title in the strip is derived: `tab.openedFile?.name ?? (tab as untitled).untitledLabel`. Two options for storing the label:
  1. Add an `untitledLabel: string | null` field to `Tab` (cleaner).
  2. Derive on the fly from tab index among untitled tabs (cheaper but flickers on close).
  Option 1 is the simpler-to-reason-about default; option 2 is fine if a contributor pushes back during review.
- The "is modified" rule for an Untitled tab is the same: `text !== savedText`. A fresh Untitled tab has empty `text` and empty `savedText`, so no asterisk until the user types.
- Closing an Untitled tab with unsaved changes goes through the standard close-confirm prompt. Choosing Save invokes Save-As; cancelling Save-As leaves the tab open with its content intact (mirrors Feature 004 §3's existing behaviour).

## 12. Out-of-scope confirmations (re-stated from spec for the tasks phase)

These are intentionally **not** addressed in this feature; planning here ensures the tasks phase does not silently expand scope:

- **Restore previous session's tabs on launch / persist tabs across launches**: out of scope. Each launch returns to the empty state. Listed as a candidate follow-up in spec Assumptions.
- **Drag-to-reorder tabs**: out of scope. Tab order is insertion order; closing a tab leaves the remaining tabs in their existing order. Listed as a candidate follow-up.
- **Detach a tab into a separate window / drag a tab between two MILF windows**: out of scope.
- **Keyboard shortcuts for tab operations (Ctrl+Tab cycle, Ctrl+W close, Ctrl+Shift+T reopen, Ctrl+1..9 jump-to-tab)**: not required by spec FR-025; candidates for a follow-up. The existing Ctrl/Cmd+S / N / O shortcuts from Feature 004 continue to act on the active tab.
- **Reload from disk / detect external file changes / file watch**: out of scope. Re-opening an already-open file re-focuses the existing tab and preserves its unsaved edits (FR-011).
- **Recent files list / recently closed tabs**: out of scope.
- **"Welcome / Open a file" CTA in the empty state**: nice-to-have, not in scope. Empty state is a clean empty editor / preview.
- **Per-tab view modes** (one tab in editor-only, another in split): out of scope. The view-mode preference is workspace-level (Feature 003).
- **Atomic write / backup file / `.bak` on save**: out of scope (carried over from Feature 004).
- **Wiring up ESLint / Prettier / test runner / CI**: still a pre-existing gap from 002 / 003 / 004; this feature does not regress it and does not address it.
