# Phase 1 — Data Model: Multi-File Editing with Tabs

This feature has no database, no on-disk schema, and no IPC payload beyond what Features 002 / 003 / 004 already use. The "data model" is the shape of in-memory React state in `App.tsx` plus the per-tab CodeMirror snapshots held in a ref. Persistence to disk happens through the existing `saveMarkdownFile` / `saveMarkdownFileAs` calls in `src/lib/fileOpen.ts`; localStorage persistence is unchanged from Feature 004.

## 1. Top-level entities

### `Tab`

A single open document in the workspace.

| Field          | Type                                          | Notes |
|----------------|-----------------------------------------------|-------|
| `id`           | `TabId` (alias for `string`, e.g. `"tab-3"`)  | Monotonic, unique within a session. Generated by `nextTabId()`. Stable for the tab's lifetime — never re-keyed when `openedFile` changes (e.g. after Save-As). |
| `text`         | `string`                                      | Current editor content for this tab. Updated on every keystroke (debounced naturally by CodeMirror's update listener). |
| `savedText`    | `string`                                      | Content last successfully written to disk for this tab, OR the content the tab was opened with if never saved this session, OR `""` for a fresh Untitled tab. Derives `isModified = text !== savedText`. |
| `openedFile`   | `{ name: string; path: string } \| null`      | When non-null, the absolute path the tab was opened from (or saved-as to) and its basename. When `null`, the tab is **Untitled** (the existing New-button path; see research.md §11). |
| `untitledLabel`| `string \| null`                              | When `openedFile === null`, the display label (e.g. `"Untitled-2"`). When `openedFile !== null`, this field is `null` and the basename is used instead. Computed at tab-creation time; never re-computed. |

**Derived (computed at render time, never stored)**:

- `isModified: boolean` = `text !== savedText`. Drives the per-tab asterisk indicator (FR-007).
- `displayName: string` = `openedFile?.name ?? untitledLabel ?? "Untitled"`. The tab title shown in the strip and in the hover tooltip.
- `hoverTitle: string` = `openedFile?.path ?? displayName`. The full path (or just the label) shown via the `title` attribute on the pill (FR-008).

**Validation rules**:

- Exactly one of `openedFile` / `untitledLabel` is non-null at any time (Untitled tabs have a label, file-backed tabs have a path).
- `id` is unique across all tabs in the current session.
- `text` and `savedText` are always strings (`""` is valid; `null`/`undefined` is not).

**State transitions** (the only mutators):

| Event                          | Effect on the Tab |
|--------------------------------|------------------|
| User types in the editor       | `text` updated. `savedText` unchanged → `isModified` may flip true. |
| Save (manual or auto) succeeds | `savedText` ← snapshot of `text` at save-kickoff time → `isModified` flips false. If the tab was Untitled and Save was via Save-As: `openedFile` ← `{ name, path }`, `untitledLabel` ← `null`. |
| Save fails                     | `savedText` unchanged. `isModified` stays true. The shared `error` state is set with a tab-attributed message. |
| Tab is closed                  | Tab is removed from `tabs[]` entirely; its `EditorState` snapshot is deleted from `editorStatesRef`. |

### `TabSet` (workspace state at the App level)

The ordered collection of all currently open tabs plus the active-tab pointer.

| Field          | Type                                | Notes |
|----------------|-------------------------------------|-------|
| `tabs`         | `Tab[]`                             | Insertion-ordered. Visible order in the strip mirrors this array. |
| `activeTabId`  | `TabId \| null`                     | The single active-tab pointer. `null` only when `tabs.length === 0` (empty state). |

**Derived (computed at render time)**:

- `activeTab: Tab \| null` = `tabs.find(t => t.id === activeTabId) ?? null`. The shorthand the rest of the App reads.
- `activeText: string` = `activeTab?.text ?? ""`. The string passed to `<Workspace />` (and onward to `<Editor />` and `<Preview />`).
- `saveEnabled: boolean` = `activeTab !== null && activeTab.openedFile !== null && !savingByTab[activeTab.id]`. Drives the Save button's `disabled` prop. **Note**: for an Untitled active tab, `saveEnabled` is `false` only if you want to force the user through Save-As explicitly via a separate path; in practice we keep `saveEnabled = activeTab !== null && !savingByTab[activeTab.id]` so the existing Save-As fallback in `performSave` is reachable for Untitled tabs too. The exact rule is documented in `contracts/components.md`.

**Invariants**:

- `tabs.length === 0` ⇔ `activeTabId === null` (no orphan pointers, no missing pointers).
- `activeTabId !== null` ⇒ `tabs.some(t => t.id === activeTabId)` (the pointer references a real tab).
- `tabs` contains no duplicate IDs (the `nextTabId()` counter is monotonic per session).
- For file-backed tabs (`openedFile !== null`), no two tabs share an `openedFile.path` (the dedup rule from FR-011).

**State transitions**:

| Event                          | Effect on TabSet |
|--------------------------------|-----------------|
| User opens a file via the dialog, no existing tab for that path | `tabs.push(newTab)`; `activeTabId ← newTab.id`. |
| User opens a file via the dialog, existing tab has that path | `tabs` unchanged; `activeTabId ← existingTab.id`. Existing tab's `text` / `savedText` NOT reloaded from disk. |
| User clicks a tab in the strip | `activeTabId ← clickedTab.id`. `editorStatesRef` snapshot/restore runs around this update. |
| User clicks New (or presses Ctrl/Cmd+N) | `tabs.push(untitledTab)`; `activeTabId ← untitledTab.id`. |
| User closes an unmodified tab | `tabs` removes the closed tab; `activeTabId ← neighbor.id` if the closed tab was active, else unchanged. |
| User closes a modified tab → chooses Save → success | Tab is saved (savedText updated); then `tabs` removes the tab; neighbor activation if needed. |
| User closes a modified tab → chooses Save → failure | `tabs` and `activeTabId` unchanged; error banner shown; modified indicator remains. |
| User closes a modified tab → chooses Discard | `tabs` removes the tab; in-memory edits lost; neighbor activation if needed. |
| User closes a modified tab → chooses Cancel | `tabs` and `activeTabId` unchanged. No save attempted. |
| Last tab closed | `tabs = []`; `activeTabId = null`. Empty state. |

### `User Preferences` (unchanged from Feature 004)

| Key in `localStorage`  | Type                                    | Default |
|------------------------|------------------------------------------|---------|
| `milf.theme`           | `"light" \| "dark"`                      | System preference; falls back to `"light"`. |
| `milf.viewMode`        | `"editor" \| "preview" \| "split"`       | `"split"`. |
| `milf.autoSave`        | `"on" \| "off"` (exposed as `boolean`)   | `false`. |

This feature does NOT add a new preference. Tabs are not persisted. The `milf.viewMode` and `milf.autoSave` keys continue to live at the workspace level (one setting shared across all tabs).

## 2. Refs (state that survives renders but does not trigger them)

### `editorStatesRef: Ref<Map<TabId, EditorState>>`

Per-tab snapshots of CodeMirror's `EditorState`. Captures the selection, undo history, and scroll snapshot for each tab so a switch-and-return preserves the user's place.

- Read on tab activation: `editorStatesRef.current.get(incomingId)` — passed to `editorHandle.setState(...)` if present.
- Written on tab deactivation: `editorStatesRef.current.set(outgoingId, editorHandle.getState())`.
- Deleted on tab close: `editorStatesRef.current.delete(closedId)`.

Map size is O(open-tab-count). Each `EditorState` is a few KB of structured CodeMirror data — not a copy of the doc string (CodeMirror's `Text` is a persistent rope shared with the live view as long as the view is alive).

### `savingByTab: Record<TabId, boolean>` (React state, not a ref)

Per-tab "save in flight" flag. Lives in React state because it drives UI (the Save button's disabled state for the active tab).

- Read by the Save button (`savingByTab[activeTabId] ?? false`).
- Read by the auto-save effect (skips scheduling if `savingByTab[activeTabId]` is true).
- Read by `performSave(tabId)` (sets the pending follow-up if true).
- Mutated by `performSave(tabId)` (set true on entry, false on completion).
- Cleared on tab close.

### `pendingSaveRef: Ref<Map<TabId, boolean>>`

Per-tab "another save is needed after the current one finishes" flag. Lives in a ref because it does not need to drive UI re-renders.

- Set to true by `performSave(tabId)` when called while a previous save for the same tab is in flight.
- Read at the tail of `performSave(tabId)` to decide whether to immediately schedule one more `performSave(tabId)` call.
- Cleared on tab close.

## 3. Tab-attributed errors

There is a single workspace-level `error: string \| null` state (unchanged shape from Feature 003 / 004) but its messages now include the failing tab's name so the user can attribute the failure (FR-021):

- Manual / auto-save failure on tab T: `setError(\`Could not save ${T.displayName}: ${friendlyMessage}\`)`.
- Open failure (no tab created): `setError(\`Could not open: ${friendlyMessage}\`)` — same shape as Feature 003 (no tab attribution needed; nothing was opened).
- Discard / Cancel from the close-confirm dialog: never sets an error.

A new save success on any tab clears the existing error via `setError(null)`. This is the same coalescing rule Feature 003 / 004 already use.

## 4. CodeMirror snapshot lifecycle (visual)

```
  User clicks tab A → B:
  ────────────────────────────────────────────────────────────────
  before:
    activeTabId = A
    editorStatesRef = { A: <EditorState_A_live_or_stale>, … }
    editor.state = <EditorState_A_live>

  step 1 (in the click handler, BEFORE setActiveTabId):
    editorStatesRef.current.set(A, editor.getState())   // refresh A
    setActiveTabId(B)                                    // schedules re-render

  step 2 (during the re-render):
    React passes value={B.text} to <Editor />
    Editor's "sync value to doc" effect: if doc !== B.text,
      dispatch transaction to replace doc.

  step 3 (useLayoutEffect keyed on activeTabId, after commit):
    const snapshot = editorStatesRef.current.get(B)
    if (snapshot) editor.setState(snapshot)              // restores selection + scroll + undo
    // else: leave editor as just-set-from-value (first activation of B)

  after:
    activeTabId = B
    editorStatesRef = { A: <fresh_snapshot_A>, B: <may_or_may_not_have_been_set>, … }
    editor.state = <EditorState_B>
```

The `useLayoutEffect` (not `useEffect`) ensures step 3 runs before the browser paints, so the user never sees the wrong-cursor frame.

## 5. What is NOT in this data model

These are intentionally excluded; trying to model them here would scope-creep the feature:

- **Recently-closed tabs** (for Ctrl+Shift+T restore): no stack of closed tabs is kept. The `editorStatesRef` entries are deleted on close.
- **Per-tab view-mode preference**: `viewMode` stays workspace-level. The TabSet model does not include a per-tab view mode.
- **MRU tab activation order**: `activeTabId` is the only pointer; there is no history stack of "previously active" IDs.
- **Tab reordering**: no `position` field; order is solely `tabs[]` insertion order.
- **Persisted tab session across launches**: the in-memory `tabs[]` is created at app startup as `[]`. No serialisation, no deserialisation.
- **Untitled-N counter persisted across sessions**: the `untitledLabel` is computed at tab-creation time from the current `tabs[]`; the counter is implicit (max-existing + 1) and resets per session because no untitled tabs survive a launch.
